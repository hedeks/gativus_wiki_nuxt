# Опыт изоляции поиска по локалям, удаления презентаций статей и исправления стилей шапки

## Описание проблемы и задачи

В ходе оптимизации публичного поиска и панели администрирования статей были выявлены три независимые проблемы:

1. **Смешение языковых результатов в поиске**: 
   * Полнотекстовый поиск (`FTS5` в SQLite) индексировал все переводы статьи или термина в рамках **одной строки** в таблице `wiki_fts` (заголовки и контент на `en`, `ru` и `zh` склеивались через пробел).
   * При поиске на русском языке (например, находясь на русской версии сайта) система находила совпадения, но если пользователь переключался на английскую версию и искал то же самое, статья всё равно находилась (так как в индексе присутствовал русский текст). На английском сайте отображался английский заголовок с обрывком русского текста в сниппете, что выглядело некорректно.
2. **Невозможность удалить презентацию статьи**: 
   * В форме редактирования статьи (`pages/admin/articles/[id]/edit.vue`) при клике на иконку «корзина» локальный стейт пути к файлу сбрасывался в `""`.
   * Однако при сохранении фронтенд отправлял данные с использованием логического ИЛИ: `presentation_path: presentationPath.value || undefined`.
   * Поскольку пустая строка приводила к отправке `undefined`, NUXT-бэкенд (`[slug].put.ts`) полностью игнорировал это поле в объекте обновления, сохраняя старое значение в базе данных.
3. **Нарушение визуального стиля кнопки поиска в шапке**: 
   * В файле `components/TheHeader.vue` были прописаны стили для унифицированного 34px прозрачного «ghost»-стиля кнопок шапки.
   * Однако стили кнопки поиска использовали селектор `.gv-header__actions--desktop :deep(.gv-search__trigger.gv-btn--chromeless)`. Так как компонент поиска `<TheSearch>` фактически расположен на левой стороне шапки внутри контейнера `.gv-header__left` (а не в `.gv-header__actions--desktop` справа), селектор не срабатывал. Кнопка поиска рендерилась с рамкой и фоном по умолчанию, выбиваясь из общего дизайна шапки Gativus.

## Проблемы, с которыми столкнулись, и их диагностика

### 1. Единый FTS-документ против многоязычности
* **Диагностика**: Поиск в SQLite FTS5 по умолчанию работает по текстовым колонкам. Если записывать все переводы в одну строку таблицы индекса, невозможно отделить совпадения на одном языке от совпадений на другом с помощью стандартного `MATCH`.
* **Решение**: Переход на модель **«одна строка индекса на одну языковую версию»**.
  В триггерах и ресинке базы данных в [migrate.ts](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/utils/migrate.ts) логика была переписана: вместо одной склеенной строки теперь для каждого перевода (если заголовок не пуст) создается отдельная запись в `wiki_fts` с соответствующим тегом локали (`locale = 'en'`, `'ru'` или `'zh'`).

### 2. Omit-поведение NUXT PUT API
* **Диагностика**: На бэкенде обновление статьи реализовано через динамическую сборку массива `updates`. Поля, пришедшие как `undefined` (отсутствующие в JSON), не включаются в SQL-запрос `UPDATE`. Для удаления (сброса значения в `NULL`) на бэкенд необходимо явно передать `null`.
* **Решение**: Изменение маппинга стейта на фронтенде с `|| undefined` на `|| null` для полей презентации при отправке запроса.

### 3. Ошибочный контекст в CSS-селекторах шапки
* **Диагностика**: Изучение разметки `TheHeader.vue` показало, что блок `.gv-header__actions--desktop` содержит только переключатель языков, смену темы и профиль. Поиск находится на десктопе в `.gv-header__left`, а на мобильных — в `.gv-header__actions--mobile`.
* **Решение**: Замена префикса селектора на общий класс поиска `.gv-header__search`, что автоматически применило прозрачный ghost-стиль к обеим кнопкам поиска (десктопной и мобильной).

## Выводы и принятые решения

### 1. Изолированный полнотекстовый поиск (SQLite FTS5)
Для изоляции поиска по языкам были переписаны триггеры обновления индекса:
* При вставке/обновлении статьи или термина старые записи для данного ID удаляются из `wiki_fts`, а затем добавляются новые строки только для тех языков, для которых заполнен заголовок:
  ```sql
  CREATE TRIGGER tr_articles_update AFTER UPDATE ON articles BEGIN
    DELETE FROM wiki_fts WHERE id = old.id AND type = 'article';
    INSERT INTO wiki_fts(id, type, title, content, slug, locale)
    SELECT new.id, 'article', new.title, new.html_content, new.slug, 'en' WHERE COALESCE(new.title, '') != '';
    INSERT INTO wiki_fts(id, type, title, content, slug, locale)
    SELECT new.id, 'article', new.title_ru, new.html_content_ru, new.slug, 'ru' WHERE COALESCE(new.title_ru, '') != '';
    INSERT INTO wiki_fts(id, type, title, content, slug, locale)
    SELECT new.id, 'article', new.title_zh, new.html_content_zh, new.slug, 'zh' WHERE COALESCE(new.title_zh, '') != '';
  END;
  ```
* Запрос к API поиска в [search.get.ts](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/api/search.get.ts) теперь жестко фильтрует по локали:
  ```sql
  SELECT wiki_fts.id AS id, wiki_fts.type AS type, wiki_fts.title AS title, wiki_fts.slug AS slug, wiki_fts.locale AS locale, ...
  FROM wiki_fts
  WHERE wiki_fts MATCH ? AND wiki_fts.locale = ?
  ```
  Это позволило отказаться от сложных `LEFT JOIN` к таблицам `articles` и `terms` при поиске, сделав запрос быстрее и проще.

### 2. Использование явного `null` для удаления опциональных параметров
* При проектировании UI-форм в Nuxt / Vue важно помнить, что пустая строка `""` или `null` должна отправляться на бэкенд без подмены на `undefined`, если целью является затирание (сброс) значения в БД. В [edit.vue](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/pages/admin/articles/%5Bid%5D/edit.vue) теперь отправляется:
  ```typescript
  presentation_path: presentationPath.value || null,
  ```

### 3. Независимость глобальных компонентов от структуры контейнера в CSS
* При стилизации общих компонентов (таких как поиск), использующих `:deep()`, предпочтительно вешать стили на собственный класс компонента (в данном случае `.gv-header__search`), а не завязываться на внешние обертки действий (`.gv-header__actions--desktop`), расположение элементов в которых может меняться при рефакторинге верстки.
