# Опыт: Частичный JSON-экспорт, проблемы кодировки заголовков и нюансы авторизации в SPA

При реализации функционала частичного экспорта статей, книг и терминов в JSON, а также предварительного просмотра графа при импорте, мы столкнулись с рядом неочевидных проблем, связанных с HTTP-заголовками и особенностями работы SPA (Nuxt 3) с JWT-авторизацией.

## 1. Ошибка 500 из-за кириллицы в заголовке Content-Disposition
**Контекст:** Для удобства пользователей мы решили генерировать динамические имена файлов, основанные на слагах терминов, фильтрах или датах (например, `term-тест-2023-10-25T14-30.json`). Заголовок для скачивания формировался стандартным образом:
```typescript
setResponseHeader(event, 'Content-Disposition', `attachment; filename="${filename}"`)
```

**Проблема:** Если переменная `filename` содержала кириллицу (что часто бывает в слагах или поисковых фильтрах), Node.js HTTP-сервер падал с исключением `TypeError [ERR_INVALID_CHAR]: Invalid character in header content ["Content-Disposition"]`. HTTP-заголовки жестко ограничены ASCII-набором символов, поэтому попытка передать туда UTF-8 символы приводила к ошибке 500.

**Решение:** Оборачивать имя файла в `encodeURIComponent` прямо внутри серверного роута:
```typescript
const encodedFilename = encodeURIComponent(filename)
setResponseHeader(event, 'Content-Disposition', `attachment; filename="${encodedFilename}"`)
```
На клиенте (фронтенде) для извлечения имени из ответа мы используем регулярное выражение с последующим декодированием:
```typescript
const contentDisposition = response.headers.get('Content-Disposition')
let filename = 'export.json'
if (contentDisposition) {
  const match = contentDisposition.match(/filename="?([^"]+)"?/)
  if (match) filename = decodeURIComponent(match[1])
}
```
Это позволило надежно передавать любые юникод-символы в именах файлов.

## 2. Ошибка 401 Unauthorized при нативном fetch
**Контекст:** Чтобы иметь возможность читать заголовки ответа (`Content-Disposition`) и получать файл как Blob, изначально использовался нативный JavaScript `fetch`:
```typescript
const headers = new Headers(store.getAuthHeader() as HeadersInit)
const response = await fetch(url, { headers })
```

**Проблема:** Запросы падали с ошибкой `401 Unauthorized`. Нативный `fetch` в среде Nuxt 3 может терять часть контекста, а также не умеет автоматически управлять некоторыми заголовками или куками, как это делает встроенный пакет `ofetch`.

**Решение:** Полный переход на встроенный в Nuxt `$fetch.raw`. Он сохраняет все возможности `ofetch` (включая консистентную передачу авторизации), но не парсит JSON автоматически, а возвращает полноценный объект `Response`, из которого можно извлечь Blob-данные и заголовки:
```typescript
const response = await $fetch.raw<Blob>(url, { 
  headers: store.getAuthHeader(),
  responseType: 'blob'
})
const blob = response._data // Само тело файла
```

## 3. "Тихое" протухание JWT-сессии в SPA и потеря данных
**Контекст:** В Single Page Application интерфейс загружается один раз. Если администратор долго находится в админке без перезагрузки вкладки, срок жизни JWT токена на сервере может истечь. Frontend об этом не знает (кука остается, статус `isLoggedIn` равен `true`).

**Проблема:** При попытке сохранить масштабные изменения или сделать экспорт после долгого простоя, запрос улетает на сервер, где токен признается недействительным. Сервер отбивает запрос (401), и пользователь теряет все введенные, но несохраненные данные, так как классической защиты от этой ситуации не было.

**Решения:**
1. **Быстрое:** Мы увеличили срок жизни токена (и соответствующей cookie) в бэкенде с 24 часов до 30 дней. Это снижает вероятность столкновения с проблемой до статистической погрешности.
2. **Архитектурное (Task of Future):** Задокументирован паттерн **Promise Queue + Auth Interceptor**. Планируется создание глобального `ofetch` интерцептора. Если приходит 401 ошибка, интерцептор не выбрасывает исключение сразу. Он «замораживает» запрос в очереди промисов и показывает всплывающее окно "Сессия истекла, введите пароль". После ввода пароля и получения нового токена, упавший запрос автоматически возобновляется. Это гарантирует нулевую потерю данных даже при истекшей сессии.
