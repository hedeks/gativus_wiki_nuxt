# Опыт реализации синхронизации поиска и пагинации с URL-адресом (Задача 36)

## Описание проблемы и задачи
На страницах списков библиотеки (`books/index.vue`), статей (`articles/index.vue`) и глоссария (`glossary/index.vue`) фильтрация (поиск, категории, алфавитный указатель) и пагинация выполняются на стороне клиента над полным массивом записей.

Для сохранения состояния фильтров при обновлении страницы или передаче ссылки использовался механизм записи параметров в URL через `navigateTo` (`?page=...&search=...&category_id=...`). 

Однако возникла проблема рассинхронизации:
* При переходах по истории браузера (кнопки «Назад» и «Вперед») URL-адрес менялся, но интерфейс (строка поиска, выбранные пиллы категорий, номер страницы пагинации) оставался в прежнем состоянии.
* В Nuxt 3 изменение query-параметров на одном и том же роуте по умолчанию не приводит к перезапуску жизненного цикла `setup()` компонента страницы. Из-за этого реактивные переменные (`ref`), инициализированные один раз при монтировании, не обновлялись вслед за URL.
* Это создавало баг, при котором поиск визуально ломался или казалось, что он ищет только по текущей открытой странице пагинации, так как после навигации состояние переменных не сбрасывалось должным образом.

---

## Проблемы, с которыми столкнулись, и их диагностика

### 1. Отсутствие обратной связи от роутера к состоянию страницы
* **Симптом**: Пользователь вводит поисковый запрос, переходит на страницу детального просмотра статьи, затем нажимает «Назад» в браузере. В URL-адресе параметры поиска стерты, но на странице всё еще отображаются отфильтрованные результаты, а в поисковом инпуте остался старый текст.
* **Причина**: Связь была односторонней: изменения `page` и `searchQuery` вели к обновлению URL, но изменения URL никак не влияли на локальные `ref` компонента.
* **Решение**: Добавление глубокого наблюдения за реактивным объектом `route.query`.

### 2. Угроза бесконечного цикла навигации (Navigation Loop Cascades)
* **Симптом**: При попытке синхронизировать URL и переменные приложение зависает, а консоль переполняется ошибками навигации Vue Router.
* **Причина**: Возникает бесконечный цикл обновления:
  1. Пользователь меняет поисковый запрос -> меняется `searchQuery`.
  2. Срабатывает watch на `[page, searchQuery]`, вызывая `navigateTo({ query: ... })`.
  3. Меняется `route.query`.
  4. Срабатывает watch на `route.query` и обновляет `searchQuery.value`.
  5. Обновление `searchQuery.value` снова триггерит watch на изменение переменных (пункт 2), запуская процесс по кругу.
* **Решение**: Внедрение строгих проверок на равенство значений перед перезаписью `ref`. Переменная обновляется только в том случае, если её текущее значение действительно отличается от пришедшего из URL.

## Выводы и принятые решения

Было реализовано централизованное решение для трех индексных страниц (`books/index.vue`, `articles/index.vue`, `glossary/index.vue`).

### 1. Синхронизирующий Watcher (`route.query` -> `Refs`)
На каждой странице развернут наблюдатель, выступающий связующим звеном при изменении URL:
```typescript
watch(
  () => route.query,
  (newQuery) => {
    // 1. Пагинация
    const p = parseInt(newQuery.page as string) || 1
    if (page.value !== p) page.value = p

    // 2. Поисковый запрос
    const s = (newQuery.search as string) || ''
    if (searchQuery.value !== s) searchQuery.value = s

    // 3. Категории (при наличии)
    const cat = newQuery.category_id ? parseInt(newQuery.category_id as string) : null
    if (activeCategory.value !== cat) activeCategory.value = cat

    // 4. Буквенный фильтр (для глоссария)
    if ('letter' in newQuery || activeLetter.value !== null) {
      const letter = (newQuery.letter as string) || null
      if (activeLetter.value !== letter) activeLetter.value = letter
    }
  },
  { deep: true }
)
```

### 2. Сброс страницы при поиске
Чтобы избежать ситуации, когда пользователь находится на 3-й странице пагинации, вводит новый поисковый запрос (под который подходит всего 2 элемента) и видит пустой экран, настроен автоматический сброс:
* При изменении `debouncedQuery` (с задержкой 300 мс) или смене категории `page.value` сбрасывается в `1`.
* Watcher на изменение переменных отслеживает это и делает переход на чистый URL без параметра `page` (поскольку `page = 1` опускается для красоты адреса).

### 3. Сохранение поддержки Infinite Scroll
Поскольку на проекте поверх стандартной пагинации реализован автоскролл через Intersection Observer, гибридная модель сохранена:
* При скролле вниз страница инкрементируется (`page.value++`), плавно подгружая новые элементы с помощью `slice(0, page.value * size)`.
* При этом URL-адрес аккуратно обновляется, сохраняя историю просмотров, а при переходе назад скролл и количество загруженных карточек восстанавливаются без рывков.
