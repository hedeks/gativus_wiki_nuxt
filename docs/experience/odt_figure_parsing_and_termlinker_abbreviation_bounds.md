# Опыт исправления парсинга ODT рисунков и автолинковщика TermLinker

В ходе этой сессии были устранены критические проблемы, связанные с импортом рисунков из файлов ODT для английского и китайского языков, а также ложные срабатывания автолинковщика глоссария (`TermLinker`).

---

## 1. Проблема

### А. Сбои парсинга рисунков (`figure`) в ODT переводах
При импорте английских и китайских ODT-файлов плейсхолдеры рисунков оставались в базе данных в виде сырого текста `[ INSERT FIGURE: ... ]` или `[ 插入插图：... ]`.
Это происходило по трем причинам:
1. **Жесткое регулярное выражение**: Парсер `odtParser.ts` ожидал строго английский формат с обязательным номером рисунка: `/\[\s*INSERT\s+FIGURE\s+(\d+)\s*[\:\-–—]?\s*.*?\s*\]/gi`. Если номер отсутствовал (например, `[ INSERT FIGURE: ... ]`) или ключевые слова были на русском/китайском, регулярка не находила плейсхолдер.
2. **Фрагментация разметки**: Редакторы типа LibreOffice при экспорте ODT оборачивали части плейсхолдера в дополнительные теги `<span>`, `<strong>` и т.д., что ломало простую блочную регулярку `blockRegex`. Также автолинковка могла сработать раньше, обернув слова внутри плейсхолдера в теги `<a>`.
3. **Отсутствие картинок в переводах**: Переводчики часто вырезают картинки из ODT-файлов перевода (в ZIP-архиве папка `Pictures/` пуста), чтобы уменьшить размер файлов. В результате у парсера не было массива `images` для подстановки.

### Б. Ложные совпадения в `TermLinker`
Аббревиатуры глоссария, состоящие из латинских заглавных букв (например, `CONT`, `MOVE`), ошибочно перелинковывали части слов в тексте (например, превращая `contour` в `<a ...>cont</a>our` или `movement` в `<a ...>move</a>ment`).
Причина заключалась в приведении ключей глоссария к нижнему регистру (`cont`, `move`) при построении Map в `buildTermsMaps`. Проверка вида `const isAllCapsLatin = /^[A-Z0-9\-_]+$/.test(phrase)` проверяла уже нижний регистр, всегда возвращала `false` и переходила в ветку обычных английских слов, допускающую суффиксы длиной до 3 символов (что позволяло сопоставить `our` из `contour`).

---

## 2. Решение

### А. Универсальный парсер рисунков и наследование медиа
1. **Гибкие регулярные выражения**:
   В [odtParser.ts](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/utils/odtParser.ts) регулярные выражения `placeholderRegex` и `blockRegex` были переписаны на поддержку:
   * Действий: `INSERT | ВСТАВИТЬ | 插入` (опционально).
   * Ключевых слов: `FIGURE | FIG | РИСУНОК | РИС | 插图 | 图片 | 图`.
   * Опциональных номеров: `(\d+)?`.
   * Произвольных тегов форматирования `(?:\s*|<[^>]+>)*` вокруг и внутри плейсхолдера.
2. **Последовательное сопоставление (Unnumbered Fallback)**:
   Если в плейсхолдере нет номера рисунка, ему автоматически назначается порядковый индекс среди всех найденных в статье плейсхолдеров (`matchIdx`). Это позволяет корректно связывать рисунки по порядку их появления в тексте.
3. **Наследование изображений из других переводов**:
   Во всех трех контроллерах импорта ODT ([bulk-odt.post.ts](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/api/import/odm/project/[id]/bulk-odt.post.ts), [part/[partId].post.ts](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/api/import/odm/project/[id]/part/[partId].post.ts), [bulk-odt-lang.post.ts](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/api/admin/books/[id]/bulk-odt-lang.post.ts)):
   * При получении запроса парсер ищет уже загруженные картинки для этой же главы в статьях на других языках в БД (например, в русском оригинале).
   * При массовом импорте картинки извлекаются из всех файлов пачки одновременно и связываются с главой (`partId`).
   * Собранный массив объединяется и передается в `parseOdtBuffer` через новую опцию `images`.

### Б. Учет оригинального регистра в `TermLinker`
1. В [termLinker.ts](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/utils/termLinker.ts) структура Map была расширена до типа `TermMeta`:
   ```typescript
   export interface TermMeta {
     id: number
     slug: string
     originalTitle: string
   }
   ```
2. Функция `tryTermMatchAt` теперь принимает `originalTitle`. Проверка аббревиатур (`isAllCapsLatin`) выполняется по оригинальному заголовку из базы данных. Если заголовок состоит только из заглавных латинских букв (например, `CONT`), суффикс жестко ограничивается: допускается только пустое окончание или буква множественного числа `s`/`S`. Это полностью исключает частичные ложные срабатывания.

## 3. Извлеченные уроки и ловушки

> [!TIP]
> **Ограничение суффиксов аббревиатур**: Если логика поиска совпадений зависит от регистра (например, для отличия аббревиатур `CONT` от обычных слов), никогда не полагайтесь на ключи Map, если они нормализуются к нижнему регистру. Всегда храните метаданные с исходным регистром (`originalTitle`).

> [!WARNING]
> **Специфика LibreOffice/Word XML**: Офисные пакеты часто дробят текст на мелкие теги `<span>` при экспорте. При написании регулярных выражений для парсинга плейсхолдеров, таких как `[ INSERT ... ]`, всегда делайте допуск на HTML-теги `(?:\s*|<[^>]+>)*` между квадратными скобками и ключевыми словами.
