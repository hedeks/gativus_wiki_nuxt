# Монолитная архитектура страницы категорий

## Проблема
Пользователь обратился с необходимостью доработать раздел категорий (`/categories`), который изначально был реализован через динамический роутинг `pages/categories/[slug].vue`. Это создавало несколько проблем:
1. **Отсутствие общей картины**: Не было единой точки входа (`/categories`), где пользователь мог бы увидеть все материалы, сгруппированные по категориям.
2. **Лишние перезагрузки**: Перемещение между отдельными категориями вызывало полную смену страницы и перерендеринг, что ухудшало UX.
3. **Ошибки маршрутизации Nitro**: Файлы API `[slug].get.ts` и `[id].put.ts` в одной директории приводили к конфликтам параметров `h3` (`Slug is required`), поскольку Radix Router перетирал `slug` параметром `id`.
4. **Тяжелые анимации**: При переключении вкладок контент улетал и прилетал издалека из-за встроенных глобальных транзишенов (`KnowledgeListTransition`).

## Решение
Для решения этих проблем была выбрана архитектура **"Монолитной страницы"**.
* Динамический роут `[slug].vue` был удален и заменен на единый `pages/categories/index.vue`.
* Для сохранения обратной совместимости старых ссылок был добавлен файл-перехватчик `[slug].vue` с использованием встроенного `definePageMeta` и middleware для выполнения 301 Permanent Redirect на новый путь с query-параметрами (`/categories?slug=...`).
* Внедрена концепция двух состояний страницы:
  1. **All Categories View**: Состояние по умолчанию (без параметров). Контент рендерится в виде древовидной структуры: крупные блоки (карточки) для родительских категорий, агрегирующие все дочерние книги, статьи и термины.
  2. **Single Category View**: Детальный вид. При выборе категории через Pills-фильтр в шапке контент переключается на вкладки (Книги, Статьи, Термины) конкретной категории.
* Тяжелые анимации `KnowledgeListTransition` были заменены на быстрые и легкие `transition` с `mode="out-in"` и коротким временем перекрестного затухания (0.2s, `cubic-bezier(0.2, 0, 0, 1)`) для создания премиального, "снэппи" (crispy) интерфейса.

## Извлеченные уроки и ловушки
1. **Конфликты роутера в Nitro/h3**: Нельзя держать файлы с разными именами динамических параметров (например, `[slug].get.ts` и `[id].put.ts`) в одной папке API. Nitro может закешировать ключ параметра и прокинуть его в `event.context.params` под неверным именем. Для решения нужно либо приводить названия файлов к единому знаменателю, либо извлекать параметр безопасно: `params.id || params.slug || Object.values(params)[0]`.
2. **Предупреждения h3 про statusMessage**: Если в `createError` передавать длинный текст или сериализованный JSON в поле `statusMessage`, `h3` будет выдавать системные предупреждения в консоли. Детальное описание ошибки необходимо класть строго в поле `message`, оставляя `statusMessage` для коротких HTTP-статусов (например, `Bad Request`).
3. **Кэширование Vite при HMR**: Если удалить Vue-компонент, который содержал блок `<style scoped>`, и создать на его месте новый без стилей (например, файл исключительно для редиректа), Vite может упасть с ошибкой `Unknown word`, пытаясь распарсить JS/TS код как CSS. Решение: добавить в новый файл пустой блок `<style scoped>/* placeholder */</style>`, либо перезапустить сервер.
4. **Оптимизация анимаций Vue**: При использовании `UTabs` из Nuxt UI старые вкладки полностью размонтируются. Использование списочных анимаций (`KnowledgeListTransition`) внутри таких вкладок приводит к проигрыванию начальных анимаций `appear` для всех элементов списка при каждом клике, что ощущается как "грязный" UX. Замена на простые `div` или быстрые `transition name="fade"` решает проблему.
