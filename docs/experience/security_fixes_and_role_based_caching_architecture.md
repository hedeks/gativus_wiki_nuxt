# Security Fixes and Role-Based Caching Architecture

## Проблема
В рамках сессии были выявлены и устранены критические уязвимости безопасности (Path Traversal при скачивании загруженных файлов и использование захардкоженного JWT-ключа в продакшене), а также внедрены базовые SWR-оптимизации для тяжелых публичных эндпоинтов (Граф Знаний). 

При обсуждении более агрессивного кэширования для статей и книг (которые обновляются раз в несколько часов) была выявлена архитектурная проблема: **выдача контента зависит от роли пользователя**. Редакторы (isEditor) должны видеть неопубликованные черновики, а обычные пользователи — нет. Это делает невозможным использование простых глобальных правил `routeRules: { swr: 3600 }` для `/api/articles/**`, так как кэширование без учета сессии приведет к утечке неопубликованного контента гостям или выдаче 404 ошибки администраторам.

## Решение текущих проблем
1. **Directory Traversal**: Использован `path.resolve(uploadsDir, param)` с последующей проверкой `filePath.startsWith(uploadsDir)`. Отказ от небезопасной очистки строк через `replace(/\.\./g, '')`.
2. **JWT Secret**: В `nuxt.config.ts` отключен фолбэк секретного ключа для `production`. Теперь при отсутствии `.env` ключа приложение выдаст 401/500 для админки, но защитит систему от подделки токенов, сохраняя при этом работоспособность публичной части сайта.
3. **SVG XSS**: Для загружаемых векторных изображений `.svg` внедрен заголовок `Content-Security-Policy: ... sandbox`, блокирующий выполнение вредоносных скриптов при их открытии напрямую.

## Извлеченные уроки и План на будущее (Role-Based Caching & Memory)

Для будущей реализации кэширования статей и других ролевых эндпоинтов необходимо соблюдать следующие архитектурные паттерны, **особое внимание уделяя утечкам памяти (Memory Leaks / OOM)**:

### 1. Кастомные обработчики с учетом ролей
Вместо `routeRules` следует использовать `defineCachedEventHandler` внутри самих API-обработчиков (например, `server/api/articles/[slug].get.ts`).
Ключ кэша должен строго включать роль пользователя:
```typescript
export default defineCachedEventHandler(async (event) => {
  // ... логика получения
}, {
  maxAge: 3600,
  getKey: (event) => {
    const role = event.context.auth?.role || 'guest'
    // Кэшируем только 2 варианта ответа на один URL: для гостей и для редакторов
    return `articles:${event.path}:role_${role}`
  }
})
```

### 2. Защита от утечек памяти (OOM - Out of Memory)
Если кэш хранится в оперативной памяти (In-Memory), возникает риск DDoS-атаки переполнением кэша. Злоумышленник может отправлять запросы с тысячами несуществующих slug'ов или уникальными query-параметрами (например, `?lang=xyz123`). Это приведет к созданию бесконечного числа записей в памяти и падению Node.js сервера (OOM).
**Способы защиты:**
- Использовать `max` лимиты в конфигурации Nitro Storage (например, драйвер `lruCache` с ограничением в 500-1000 записей). При превышении лимита старые записи будут вытесняться.
- Альтернатива: перенести кэш на диск (`driver: 'fs'`) для тяжелых ответов, чтобы память не расходовалась вообще. С диска (SSD) отдача всё равно будет в сотни раз быстрее, чем компиляция графа из базы данных.
- Строго валидировать параметры (особенно `?lang=...`) и выбрасывать 400 Bad Request ДО попадания запроса в функцию генерации кэша, если язык не входит в список `['en', 'ru', 'zh']`.

### 3. Инвалидация кэша (Cache Invalidation)
При кэшировании на 1 час редакторы должны видеть свои изменения мгновенно. В методах мутации (`PUT /api/articles/[slug]`) необходимо добавить вызов инвалидации:
```typescript
const storage = useStorage('cache')
await storage.removeItem(`nitro:handlers:articles:/api/articles/${slug}:role_editor`)
await storage.removeItem(`nitro:handlers:articles:/api/articles/${slug}:role_guest`)
```

### 4. Клиентское кэширование
Помимо бэкенда, для минимизации нагрузки следует использовать возможности клиента (Pinia Store) для хранения уже загруженных данных статьи при SPA-навигации между страницами (чтобы кнопка "Назад" отрабатывала мгновенно без сетевых запросов).
