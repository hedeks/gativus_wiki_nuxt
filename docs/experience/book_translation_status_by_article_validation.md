# Привязка индикаторов перевода книг к ручной валидации статей администратором

Этот документ описывает изменения и опыт, полученный при переработке логики статус-индикаторов перевода книг в административной панели.

---

## 1. Проблема

Ранее статус-индикаторы готовности перевода книг (`all_translated_en/ru/zh` на бэкенде) и отдельных глав проекта (`has_translation_en/ru/zh` на фронтенде) ориентировались исключительно на физическое наличие непустого HTML-содержимого статей в базе данных (`html_content IS NOT NULL AND html_content != ''`). 

Это приводило к следующим ограничениям:
1. Автоматический импорт из ODT-файлов сразу зажигал индикаторы перевода, даже если текст требовал вычитки, корректировки или содержал технический мусор.
2. Администратор не мог вручную управлять лампочками готовности перевода книги на уровне списка книг, несмотря на то, что в редакторе статьи присутствовали интерактивные кнопки переключения валидности перевода (`EN`, `RU`, `ZH`).
3. Возникал диссонанс: статья помечена администратором как не переведенная (красный флаг), а вся книга светилась как полностью готовая.

---

## 2. Решение

Было принято решение перевести расчет всех статус-индикаторов перевода книги (и на уровне всей книги, и на уровне отдельных глав) на использование полей `translation_valid_en`, `translation_valid_ru` и `translation_valid_zh` таблицы `articles`.

### А. Изменение API структуры глав проекта
В эндпоинте [[id].get.ts](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/api/import/odm/project/%5Bid%5D.get.ts):
- В SELECT-запрос из таблицы `articles` добавлены колонки `translation_valid_en`, `translation_valid_ru`, `translation_valid_zh`.
- Логика проверки готовности конкретного языка для главы изменена с проверки `r.html_content` на булеву проверку полей `translation_valid_*` (с учетом того, что для английского языка значение по умолчанию `true`/`1`):
  ```typescript
  p.has_translation_en = rows.every(r => r.translation_valid_en !== null && r.translation_valid_en !== undefined ? Boolean(r.translation_valid_en) : true)
  p.has_translation_ru = rows.every(r => Boolean(r.translation_valid_ru))
  p.has_translation_zh = rows.every(r => Boolean(r.translation_valid_zh))
  ```

### Б. Изменение SQL-запросов списка книг и деталей книги
В эндпоинтах [index.get.ts (список)](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/api/books/index.get.ts) и [[slug].get.ts (детали)](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/api/books/%5Bslug%5D.get.ts) подзапросы вычисления `all_translated_*` были переписаны.

Вместо проверки на пустоту HTML-строки:
```sql
WHERE a.id IS NULL OR a.html_content_ru IS NULL OR a.html_content_ru = ''
```

Внедрена проверка на сброшенный флаг валидности с использованием `COALESCE` для обработки `NULL` в соответствии с дефолтными значениями БД:
* Для английского (`EN`):
  ```sql
  WHERE a.id IS NULL OR COALESCE(a.translation_valid_en, 1) = 0
  ```
* Для русского (`RU`):
  ```sql
  WHERE a.id IS NULL OR COALESCE(a.translation_valid_ru, 0) = 0
  ```
* Для китайского (`ZH`):
  ```sql
  WHERE a.id IS NULL OR COALESCE(a.translation_valid_zh, 0) = 0
  ```

---

## 3. Извлеченные уроки и нюансы

1. **Особенности SQLite с булевыми значениями и json_each**:
   В SQLite нет нативного типа `BOOLEAN` — используется `INTEGER` (0 или 1). При агрегации в подзапросах через `json_each` (разворачивающих массив `imported_article_ids`) проверка вида `COALESCE(a.translation_valid_ru, 0) = 0` гарантирует, что если хотя бы у одной статьи флаг равен `0` (или запись не найдена / `NULL`), то вся глава и, соответственно, книга будут помечены как «не переведенные» (значение `0`).

2. **Дефолтные значения (Default values)**:
   При миграции схемы базы данных поле `translation_valid_en` получило значение по умолчанию `1` (так как базовый язык статей всегда считается валидным при создании), а `translation_valid_ru` и `translation_valid_zh` получили значение `0` (требуется явная ручная валидация). Использование `COALESCE(..., 1)` для EN и `COALESCE(..., 0)` для RU/ZH в SQL-запросах предотвращает неверное толкование пустых (NULL) значений у старых записей.

3. **Связь реактивного состояния фронтенда**:
   Компонент `AdminBookForm.vue` вычисляет готовность всей книги на клиенте на основе свойства `has_translation_*` каждого слота. Передача полей `translation_valid_*` из бэкенд эндпоинта структуры глав позволила обеспечить моментальное обновление статуса книги в шапке редактора при переключении лампочек в дочерней форме статьи без необходимости полной перезагрузки страницы.
