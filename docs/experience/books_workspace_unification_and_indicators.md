# Опыт рефакторинга Books Workspace: Единое рабочее пространство, обход коллизий SQLite и индикаторы перевода

## Введение и архитектурная концепция

В рамках текущей сессии была проведена масштабная реорганизация админ-панели управления книгами. Ранее логика была разделена между формой редактирования метаданных книги (`pages/admin/books/[id]/edit.vue`) и отдельной сложной страницей импорта структуры ODM/ODT (`pages/admin/import-odm.vue`). Это усложняло работу контент-менеджеров и приводило к рассинхронизации данных.

### Что было сделано глобально:
1. **Единый Workspace**: Мы полностью удалили страницу `/admin/import-odm`. Весь её функционал (парсинг ODM, загрузка ODT-файлов глав, массовый импорт, управление слотами) был интегрирован непосредственно во вкладку «Структура глав» нового универсального компонента `AdminBookForm.vue`.
2. **Бесшовный редактор контента**: В оглавление структуры добавлена кнопка открытия полноэкранного редактора статьи. Он рендерит дочерний компонент `<AdminArticleForm>` поверх формы книги, позволяя автору мгновенно перейти к WYSIWYG-редактированию текста, формул и презентаций статьи, а затем вернуться к книге в один клик без потери контекста.
3. **Автоматическая синхронизация**: Все изменения в структуре или файлах глав автоматически генерируют событие `@refresh-list`, заставляя сайдбар-список книг мгновенно обновлять своё состояние и статус-индикаторы.

## 1. Решение ошибок и инфраструктурных багов

### А. Ошибка считывания тела Blob при скачивании шаблона
* **Проблема**: При попытке скачать JSON-шаблон импорта книг фронтенд падал с ошибкой `Response.blob: Body has already been consumed`. Причиной было двойное чтение потока ответа: библиотека `$fetch` при указании `responseType: 'blob'` считывала его самостоятельно, после чего повторный вызов `await response.blob()` вызывал сбой.
* **Решение**: Мы заменили вызов метода чтения на прямое обращение к свойству `_data` объекта ответа:
  ```typescript
  const blob = response._data
  ```

### Б. Ошибки SQLite Unique Constraints в `book_categories`
* **Проблема**: При сохранении новой книги происходил сбой `NOT NULL constraint failed: book_categories.book_id`. Бэкенд пытался получить ID созданной книги через `SELECT last_insert_rowid()`, но из-за асинхронности SQLite в пул-соединениях функция возвращала некорректный ID (или `0`/`undefined`), что ломало связывание книги с её категориями.
* **Решение**: Заменили `last_insert_rowid()` на гарантированно точную выборку ID по уникальному сглаженному URL-слагу книги:
  ```typescript
  const row = await db.prepare('SELECT id FROM books WHERE slug = ?').get(slug)
  const bookId = Number(row.id)
  ```

### В. Выбор неактуального проекта ODM
* **Проблема**: Если для книги ранее по ошибке или в ходе экспериментов было создано несколько записей в `odm_projects`, API отдавал случайный проект, из-за чего структура глав ломалась или отображалась пустой.
* **Решение**: В эндпоинте [[slug].get.ts](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/api/books/[slug].get.ts) добавлена явная сортировка по убыванию ID:
  ```sql
  SELECT id FROM odm_projects WHERE book_id = ? ORDER BY id DESC LIMIT 1
  ```
  Это гарантирует выбор последней (самой актуальной) структуры.

## 2. Управление структурой и импортом

### А. Разрешение Drag & Drop сортировки глав
* **Проблема**: Ранее изменение порядка глав в скелете было заблокировано, если книги уже имели импортированный контент. Мы сняли это ограничение, разрешив свободное перетаскивание глав в любой момент.
* **Обход коллизий индексов**: SQLite прерывал пошаговый апдейт `sort_order` из-за уникального индекса `UNIQUE(project_id, sort_order)`. Для обхода ограничения мы применили двухэтапную транзакцию:
  1. Переносим все перемещаемые слоты во временную отрицательную зону (`sort_order = -(index + 1)`), освобождая положительный диапазон.
  2. Присваиваем слотам правильные положительные индексы по порядку.
  3. Каскадно пересчитываем и перезаписываем `sort_order` всех привязанных статей в таблице `articles` для сохранения порядка на сайте.

### Б. Воссоздание структуры из сиротских статей
* **Реализация**: В случае, когда статьи у книги в БД есть, а проекта структуры нет (например, книга создана старыми методами), мы выводим предупреждение и кнопку «Собрать структуру». На бэкенде в `project.post.ts` опция `is_from_existing: true` находит все статьи книги, генерирует для них слоты со статусом `imported` и прописывает в них `imported_article_ids`.

### В. Ручная привязка существующих статей
* **Реализация**: На бэкенде добавлена поддержка `article_id` при создании слота главы. На фронтенд добавлено модальное окно с поиском статей, позволяющее связать любую уже существующую статью в базе данных со свободным слотом в структуре книги.

### Г. Интерактивная сортировка книг в каталоге
* **Реализация**: Числовой инпут `sort_order` в форме книги заменен на селектор `USelectMenu` с опциями («Не менять», «Вставить в конец», «Вставить перед [Название книги]»). Бэкенд на основе выбранного `sort_position` автоматически высчитывает правильное место и сдвигает остальные книги.

## 3. Неразрушающее удаление глав (слотов)

* **Проблема**: Удаление одной главы вызывало каскадный сброс (`cascadeResetOdmSlotsFrom`) этой главы и всех глав после неё, полностью стирая ODT-файлы и статьи последующих глав. Если хотя бы одна последующая глава была опубликована, операция намертво блокировалась проверкой безопасности, не давая удалить даже черновик.
* **Решение**: Логика удаления в [[partId].delete.ts](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/server/api/import/odm/project/[id]/part/[partId].delete.ts) полностью переписана:
  - **Для опубликованных статей**: Если статья удаляемой главы опубликована (`is_published = 1`), мы её не удаляем с сайта. Вместо этого сбрасывается её привязка к книге: `book_id = NULL, sort_order = 0`.
  - **Для черновых статей**: Черновые варианты статей удаляемой главы стираются из базы.
  - **Последующие главы**: Их файлы и тексты остаются нетронутыми, выполняется лишь сдвиг их индексов `sort_order = sort_order - 1`.
  - **Синхронизация порядка**: Оставшиеся статьи книги последовательно переиндексируются для сохранения сплошного оглавления.

## 4. Автоматическая индикация перевода (Лампочки EN, RU, ZH)

* **Алгоритм**: Книга считается полностью переведенной на конкретный язык, если у неё создан проект структуры и **для каждой включенной в импорт главы (`is_enabled = 1`) успешно загружен файл ODT перевода на этот язык**.
* **Реализация бэкенда**: В SQL-запросы выгрузки книг добавлены подзапросы, сравнивающие общее число включенных глав книги с числом глав, у которых пути ODT-файлов не равны NULL:
  ```sql
  (
    SELECT CASE 
      WHEN COUNT(p.id) = 0 THEN 0 
      WHEN SUM(CASE WHEN p.odt_storage_path_zh IS NOT NULL THEN 1 ELSE 0 END) = COUNT(p.id) THEN 1 
      ELSE 0 
    END
    FROM odm_project_parts p
    WHERE p.project_id = (SELECT id FROM odm_projects WHERE book_id = b.id ORDER BY id DESC LIMIT 1) AND p.is_enabled = 1
  ) as all_translated_zh
  ```
* **Реализация фронтенда**:
  - В левом списке книг под названием каждой книги отображаются мини-шильдики `EN`, `RU`, `ZH`. При полной готовности перевода шильдики подсвечиваются ярким цветом, иначе остаются блеклыми.
  - В шапке редактора книги выводятся аналогичные индикаторы. Для мгновенного отклика на действия пользователя они вычисляются реактивно на основе массива слотов.
