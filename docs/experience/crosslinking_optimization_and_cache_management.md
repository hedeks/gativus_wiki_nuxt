# Оптимизация алгоритма перелинковки терминов и управление серверным кешем

## Проблема

1. **Производительность и точность алгоритма автоматической перелинковки:**
   * Алгоритм перелинковки терминов в HTML (`linkTermsInHtml`) работал недостаточно эффективно и давал ложные срабатывания. 
   * Он проверял совпадения посимвольно (на каждом символе текста) и создавал новые объекты `RegExp` внутри цикла.
   * Ограничение длины суффиксов (`suffixLen > 3`) было слишком нестрогим. Из-за этого происходили ложные связывания совершенно разных слов со схожим началом (например, короткое слово `cont` линковало `contour`, `move` линковало `movement`).
   * В многоязычных статьях (особенно на русском или китайском) английские термины и аббревиатуры не связывались автоматически, если для них не был прописан явный перевод, что заставляло пользователей дублировать английское название в поля `title_ru` и `title_zh`.
2. **Ограничения релинкинга:**
   * Функция «Перелинковать базу» обрабатывала абсолютно все статьи. Если изменялась одна статья, запускать глобальную перелинковку было долго и ресурсозатратно.
3. **Сброс Role-Based Caching:**
   * Ранее был внедрен кеш `lruCache` для API-эндпоинтов, который спасал сервер от OOM (Out Of Memory) при тяжелых запросах. Однако у администраторов не было удобного UI-инструмента для полного сброса этого кеша в случае рассинхронизации или после массовых изменений базы.

## Решение

### Улучшение алгоритма перелинковки
1. **Предкомпиляция и флаг `y` (sticky):**
   * Регулярные выражения для терминов теперь предкомпилируются (создаются один раз перед обходом документа).
   * Использован флаг `y` (`iy`), что заставляет RegExp искать совпадение строго с текущей позиции (`regex.lastIndex = start`), исключая медленное слайсирование строк (`text.slice(start)`).
2. **Пропуск символов внутри слов:**
   * Добавлена проверка `WORD_BOUNDARY_BEFORE.test(segment[i - 1])`. Если предыдущий символ является буквой, цифрой или другим символом слова (включая символы из кириллицы), алгоритм пропускает весь цикл проверок фраз и просто идёт дальше. Попытка наложения регулярного выражения происходит **только** на границах слов. Для китайских иероглифов, которые не входят в `WORD_BOUNDARY_BEFORE`, алгоритм корректно сохраняет посимвольную проверку.
3. **Языковой Fallback в картах терминов:**
   * В функции `buildTermsMaps` при формировании словарей `ru` и `zh` теперь всегда добавляется английский `title` и `slug` в качестве резервных ключей.
4. **Строгая валидация окончаний (суффиксов):**
   * Логика суффиксов переведена с подсчета длины на точные проверки по регулярным выражениям:
     - Для английских слов (`isPureLatin`): разрешены только окончания `^(s|es|ed|ing|er|ly)?$`.
     - Для русских слов (склонения): разрешены только `^(а|у|е|ом|ой|и|ов|ам|ами|ях|ям|ев|ей|ый|ого|ому|ым|ая|ую|ое|ые|ых|ыми|ь|я|ю|ем|ью|мя)?$`.
     - Для аббревиатур (CAPS): разрешено только добавление множественного числа `^[sS]?$`.

### Точечный релинкинг статьи
Создан эндпоинт `POST /api/admin/relink-article/[id]`, который пересобирает HTML-контент и пересчитывает граф (таблицу `article_terms`) только для одной статьи. В UI-компонент редактора (`AdminArticleForm.vue`) добавлена соответствующая кнопка "Перелинковать", позволяющая обновить связи "на лету" без затрагивания остальной базы.

### Очистка кеша сервера из UI
Создан эндпоинт `POST /api/admin/cache/clear`, вызывающий метод `useStorage('cache').clear()` у хранилища Nitro (unstorage). На странице Синхронизации (`pages/admin/sync.vue`) добавлена отдельная вкладка с красной кнопкой для сброса кеша, что дает администраторам полный контроль над инвалидацией.

## Извлеченные уроки и ловушки

- **Stateful Regex в JavaScript:** При использовании флагов `g` (global) или `y` (sticky) объект `RegExp` сохраняет состояние (`lastIndex`). Важно сбрасывать `regex.lastIndex = start` **перед каждым** вызовом `exec()`, даже если предыдущий вызов вернул `null`.
- **TypeScript Union Types в `ref`:** При расширении существующих интерфейсов (добавлении новой вкладки `cache` в UI) необходимо внимательно проверять типы `ref<'export' | 'import'>`. Вставка значения, не описанного в union, приводит к сбою сборки Nuxt-приложения.
- **Посимвольный парсинг vs регулярные выражения:** Простой `if`-пропуск (проверка предыдущего символа по словарю `WORD_BOUNDARY_BEFORE`) внутри `while`-цикла позволяет в десятки раз ускорить работу алгоритма автолинковки, избегая вызова тяжелых regex-проверок для каждой буквы текста.
