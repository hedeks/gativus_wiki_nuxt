# Опыт разработки единого рабочего пространства глоссария, эго-графа связей и кастомных селекторов фильтрации

## Описание проблемы и задачи

В рамках улучшения администрирования проекта Gativus была поставлена задача отказаться от раздельных страниц просмотра, редактирования и создания терминов глоссария в пользу **единого двухпанельного Workspace** по маршруту `/admin/glossary`. 

При реализации этого функционала возникли следующие технические вызовы и баги:

1. **Баг сброса маршрута при `term_id=0`**:
   * Изначально для создания нового термина использовался числовой идентификатор `0` (адрес `/admin/glossary?term_id=0`).
   * В Nuxt-роутере из-за проверок на приведение типов и истинность значения (truthiness) значение `0` распознавалось как falsy, в результате чего параметр запроса удалялся из URL, сбрасывая страницу в начальное состояние выбора термина.
   * Если же страница всё-таки пыталась загрузить `term_id=0`, бэкенд возвращал ошибку 404 (поскольку термина с ID `0` в базе данных не существует).
2. **Специфичность CSS для селектов в выпадающих панелях (Teleport)**:
   * Фильтры списка терминов используют обертку `<ExpandableFilters>`, которая рендерит контент через портал Nuxt/Vue (`<Teleport>`) на уровень `body`.
   * Из-за этого стандартные стили scoped-компонентов переставали работать, и селекты внутри фильтров перезаписывались глобальными стилями форм из `app.vue`.
3. **Отсутствие валидации и баг пустых статей**:
   * При создании статьи через форму не проверялось наличие обязательных полей (названия и определения на английском языке EN), что могло приводить к сохранению битых записей.
   * Сохранение термина без контента статьи ошибочно генерировало пустую запись статьи в БД, переводя форму в режим редактирования статьи, хотя пользователь её не создавал.
4. **Необходимость контекстного графа**:
   * Для каждого термина в правой панели требовалось отобразить интерактивную визуализацию его связей (в каких статьях упоминается, к каким категориям относится) без перенаправления на основную страницу графа.
5. **Массовые операции и многокритериальная сортировка**:
   * Требовалось реализовать множественный выбор терминов с помощью чекбоксов и групповое удаление с каскадным очищением связанных данных.
   * Требовалось выводить в списке количество упоминаний термина в статьях и реализовать сортировку по популярности (количеству упоминаний) и по названиям категорий.

## Диагностика и технические решения

### 1. Переход на строковый плейсхолдер `term_id=new`
* **Диагностика**: Число `0` в JavaScript/Vue часто ведет к логическим ошибкам при проверках вида `if (id)`. В `pages/admin/glossary/index.vue` и `components/admin/WorkspaceEditor.vue` логика синхронизации параметров роута сбрасывала `0`, а бэкенд падал при запросе `/api/admin/terms/0`.
* **Решение**: Было принято решение использовать явный строковый маркер `'new'` (например, `/admin/glossary?term_id=new`).
  * Во фронтенд-компонентах и роутах добавлена проверка: если `term_id === 'new'`, WorkspaceEditor переходит в режим создания (`Create Mode`), очищает форму, а кнопка «Сохранить» отправляет запрос `POST /api/terms` вместо `PUT`.
  * Маршрут `pages/admin/glossary/create.vue` был обновлен так, чтобы на входе перенаправлять пользователя на `/admin/glossary?term_id=new` вместо `term_id=0`.

### 2. Стилизация телепортируемых фильтров с использованием `!important`
* **Диагностика**: Разметка `<ExpandableFilters>` переносится в конец DOM-дерева страницы. Из-за этого scoped-стили родительской страницы не применялись к выпадающему списку селектов.
* **Решение**: Добавлены unscoped-правила с селектором `.filters-dropdown-inner .gv-admin-filter-select` и флагами `!important` в [ExpandableFilters.vue](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/components/common/ExpandableFilters.vue) для перезаписи глобальных стилей `app.vue`. Селекты получили фирменное оформление Gativus (кастомные стрелочки, отступы, фоны, радиусы скругления в темной и светлой темах).

### 3. Валидация на фронтенде и условное сохранение статьи
* **Диагностика**: В режиме создания термина бэкенд ожидает обязательные английские поля. В WorkspaceEditor не хватало клиентской валидации. Также, если в полях статьи не было контента (пустой WYSIWYG редактор), форма отправляла параметры статьи, из-за чего создавалась пустая статья.
* **Решение**:
  * В метод `save()` добавлены проверки:
    ```typescript
    if (!form.title || !form.title.trim()) { ... }
    if (!form.definition || !form.definition.trim()) { ... }
    ```
    При нарушении выводится Toast-уведомление с ошибкой.
  * Добавлена логика условной отправки контента статьи: параметры `html_content` передаются в запросе только в том случае, если пользователь действительно заполнил хотя бы один язык или прикрепил презентацию.

### 4. Доработка Knowledge Graph для фокусного режима (Ego Graph)
* **Диагностика**: Компонент `KnowledgeGraphVisualizer` инициализировал граф в масштабе по умолчанию и не центрировал его на конкретной вершине.
* **Решение**:
  * В [KnowledgeGraphVisualizer.vue](file:///c:/Users/sv653/Desktop/coding/gativus-wiki-nuxt/components/KnowledgeGraphVisualizer.vue) добавлен входной параметр `initialFocusNodeId` и watcher для него.
  * При изменении значения watcher находит нужный узел в симуляции D3 и принудительно переключает визуализатор в фокусный режим (Focus Mode / Ego Graph), подсвечивая связи выбранного термина.
  * В WorkspaceEditor встроена кнопка «Показать связи узла», которая открывает полноэкранный `UModal` с встроенным графом, отфильтрованным по языку редактирования.

### 5. Полноэкранный блокирующий лоадер при операциях над БД
* **Диагностика**: Операции перелинковки всех статей (`/api/admin/relink`) и восстановления разметки (`/api/admin/repair-linker-html`) выполняются асинхронно на сервере и могут занимать до 30-60 секунд. Отсутствие блокировки интерфейса в этот момент позволяло пользователю совершать другие действия, что могло привести к коллизиям в БД.
* **Решение**: Добавлен полноэкранный модал `<UModal v-model="isToolsModalOpen" prevent-close>` с анимацией загрузки и текстовым описанием шагов, полностью блокирующий интерфейс до завершения фонового процесса.

### 6. Групповые операции и множественный выбор терминов
* **Диагностика**: Управление списком по одному элементу при большом числе терминов неэффективно и неудобно для модератора.
* **Решение**: Добавлен стейт `selectedIds` типа `Set<number>`. В верстке списка добавлены чекбоксы для каждого элемента и верхний тулбар "Выбрать все". На основе выделения динамически рассчитывается и отображается кнопка удаления, которая шлет последовательные `DELETE` запросы и инвалидирует кэш списка терминов через `refreshNuxtData`.

### 7. Подсчет упоминаний и расширенная сортировка на бэкенде
* **Диагностика**: По умолчанию список сортировался только по алфавиту названия термина. Сортировка по категориям или числу упоминаний в статьях отсутствовала, так как в основной таблице `terms` нет физического поля количества упоминаний.
* **Решение**:
  * SQL-запрос в `server/api/terms/index.get.ts` был оптимизирован с помощью подзапроса `(SELECT COALESCE(SUM(mention_count), 0) FROM article_terms WHERE term_id = t.id) as mention_count` для динамического вычисления популярности.
  * Были добавлены SQL-параметры `orderSql` для поддержки сортировок: `mentions_desc`/`mentions_asc` и `category_asc`/`category_desc`. Для корректной сортировки по категориям использовались конструкции `CASE WHEN ? = 'ru' AND c.title_ru IS NOT NULL ...` с учетом активного языка локализации.

## Выводы и рекомендации для будущей разработки

1. **Использование строковых констант вместо `0` в URL**:
   При проектировании SPA/Nuxt-приложений избегайте передачи `0` в качестве ID нового ресурса в query-параметрах, если этот параметр участвует в проверках на существование. Рекомендуется использовать строковый маркер вроде `new` или `create`.
2. **Изоляция Teleport-стилей**:
   Любые компоненты UI-библиотек, использующие рендер через порталы (модальные окна, выпадающие списки, подсказки), требуют глобальных (unscoped) стилей или CSS-переменных, так как стандартные scoped-стили к ним не доходят.
3. **Фронтенд-валидация перед тяжелыми запросами**:
   Обязательные поля схемы БД должны быть продублированы валидацией на фронтенде с выводом понятных пользователю уведомлений до отправки сетевого запроса.
4. **Оптимизация вычисляемых полей в СУБД**:
   Для динамических счетчиков (таких как `mention_count`) использование агрегирующих подзапросов в SQLite является простым и гибким решением на средних объемах данных. При росте базы данных до сотен тысяч записей потребуется перенести подсчет упоминаний в триггеры и кэшировать это значение в виде колонки в таблице `terms`.
