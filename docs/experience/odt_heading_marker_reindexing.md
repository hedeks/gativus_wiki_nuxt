# Опыт исправления перенумерации ODT-заголовков (odt-heading-marker) в админ-панели

## Описание проблемы
При попытке переиндексации (изменения стартового номера) глав статьи в админ-панели Nuxt 3 некорректно работала функция обновления номеров глав и подразделов в разметке ODT. В частности:
1. Номера разделов первого уровня (например, `<span class="odt-heading-marker">Глава 3.</span>` или `第1章 `) и подразделов (например, `1.  1. ` или `1章  1. `) не изменялись при клике на перенумерацию.
2. В интерфейсе выбора локалей внутри модального окна переиндексации возникала ошибка JavaScript в консоли браузера (`TypeError: Assignment to constant variable`), из-за чего переключение формата нумерации переставало функционировать.

## Исследование и диагностика
Мы проанализировали исходный код утилиты переиндексации заголовков и фронтенд-компонент:
1. **Регулярные выражения в `server/utils/reindexHeadingMarkers.ts`:**
   * Детектирование глав 1-го уровня выполнялось массивом `LEVEL1_PATTERNS` (например, `/^Глава\s+\d+$/`), а подуровней — выражением `const SUBLEVEL_RE = /^(\d+)((?:\.\d+)+)$/`.
   * В реальных файлах ODT, экспортированных из LibreOffice/Word, маркеры оглавления часто содержат знаки препинания на конце (`.`, `:`), например: `Глава 3.`, `1.1.`, `Chapter 10:`.
   * Китайские маркеры вложенности генерируются в формате `1章  1.` (где присутствует иероглиф `章` и множественные пробелы).
   * Исходные регулярные выражения ожидали строгое отсутствие нецифровых символов (кроме точек) и падали на любой точке на конце, двоеточии или иероглифе.
2. **Пробелы и форматирование:**
   Строка перед разбором очищалась через `.trim()`, из-за чего терялись исходные пробелы перед закрывающим тегом `</span>` (например, `1.  1. ` превращался в `2.  1.`, сжимая верстку).
3. **Реактивность во фронтенде (`pages/admin/articles/[id]/edit.vue`):**
   При рендере списка локалей в цикле `v-for` использовались мутаторы:
   ```javascript
   { col: 'html_content', label: 'EN', model: reindexLocaleEn, setter: (v: any) => reindexLocaleEn = v }
   ```
   Так как `reindexLocaleEn` объявлен как `const` Ref в `<script setup>`, попытка прямого переприсваивания ссылки вызывала фатальную ошибку JS в браузере.

## Выводы и решение

### 1. Исправление бэкенда (`server/utils/reindexHeadingMarkers.ts`)
* **Обобщение регулярного выражения вложенности (`SUBLEVEL_RE`):**
  Мы переписали регулярное выражение для подразделов на более гибкое:
  ```typescript
  const SUBLEVEL_RE = /^(\d+)([^\d]+\d+.*)$/
  ```
  Оно захватывает первую группу цифр (номер главы), а затем разрешает любые нецифровые символы (пробелы, точки, иероглифы вроде `章`) перед последующими цифрами.
* **Сохранение пробелов и хвостовых знаков:**
  Перед сопоставлением мы извлекаем исходные начальные/конечные пробелы (`leadingSpace`, `trailingSpace`) и хвостовые знаки препинания (`separator`):
  ```typescript
  const leadingSpace = raw.match(/^\s+/)?.[0] || '';
  const trailingSpace = raw.match(/\s+$/)?.[0] || '';
  const text = raw.trim();
  const separator = text.match(/[\.:]+$/)?.[0] || '';
  const cleanText = separator ? text.slice(0, -separator.length).trim() : text;
  ```
  После форматирования новой строки мы склеиваем их обратно:
  `leadingSpace + replacement + separator + trailingSpace`.

### 2. Исправление фронтенда (`pages/admin/articles/[id]/edit.vue`)
* Сеттеры в модальном окне переиндексации обновлены для работы через `.value`:
  ```typescript
  setter: (v: any) => reindexLocaleEn.value = v
  ```
  Это восстановило реактивность и устранило JS-ошибку перезаписи константы.

Все тест-кейсы для русского, английского и китайского языков (включая сложные вложенные структуры) успешно проходят верификацию.
