# Концепт: Storytelling в Графе Знаний

## 1. Роль и Use Cases (Зачем это нужно?)
В текущей архитектуре графа (где связи строятся на основе терминов, упомянутых в книгах, главах и статьях) пользователь может легко потеряться при свободном исследовании. Storytelling решает эту проблему, предлагая **управляемое погружение (guided tour)**.

**Ключевые Use Cases:**
1. **Онбординг (Introduction Tour):** Для новых пользователей — проведение по основным узлам вики (например: "Откуда берется концепт X" -> "Главная книга Y" -> "Важный термин Z").
2. **Образовательные сценарии:** Изучение конкретной предметной области шаг за шагом с подсветкой того, как термины связаны между собой.
3. **Презентационный режим:** Демонстрация сложности и связанности базы знаний инвесторам/коллегам.

## 2. Архитектура Машины Состояний (FSM)
Для обеспечения детерминированности и защиты от багов интерфейса (когда D3 граф продолжает симулировать физику, пока играет анимация, или пользователь кликает куда-то не туда), режим сторителлинга должен строиться как строгая машина состояний.

### Возможные состояния (States):
- `IDLE`: Обычный режим работы графа.
- `STORY_START`: Инициализация презентации, блокировка лишнего инпута, подготовка UI (появление полосы прогресса, скрытие лишних меню).
- `TRANSITIONING`: Анимация (камера центрируется на нужном узле графа, D3 плавно перемещает zoom/pan, "рисуется" ребро от предыдущего узла к новому). 
- `PLAYING_NODE`: Камера зафиксирована на узле. Открывается popup/side-panel с контентом (заголовок, краткое описание термина/книги из БД).
- `PAUSED`: Пользователь остановил презентацию, чтобы осмотреться вручную (возможность временного unlock'а зума).
- `STORY_END`: Сводка пройденного пути, кнопка возврата в `IDLE`.

### Переходы (Transitions):
- `IDLE` -> (событие `play_story`) -> `STORY_START` -> `TRANSITIONING`
- `TRANSITIONING` -> (событие `anim_end`) -> `PLAYING_NODE`
- `PLAYING_NODE` -> (событие `next_step`) -> `TRANSITIONING`
- `PLAYING_NODE` -> (событие `pause`) -> `PAUSED`
- `PAUSED` -> (событие `resume`) -> `TRANSITIONING` / `PLAYING_NODE`
- `PLAYING_NODE` / `PAUSED` -> (событие `stop`) -> `STORY_END` -> `IDLE`

## 3. Уровни сложности реализации (Декомпозиция)

### Уровень 1: MVP (Жестко заданный маршрут + Простая камера)
- Захардкоженный массив ID узлов (путь).
- FSM управляет только центрированием камеры (zoomToNode) по массиву.
- Рядом с графом появляется простое окно с текстом (заглушки или статика).
- Блокировка пользовательского взаимодействия с графом во время проигрывания.

### Уровень 2: Интерактивный FSM с подгрузкой данных (Динамика)
- Маршрут задается JSON-объектом, содержащим не только узлы, но и тайминги/тексты.
- В состоянии `PLAYING_NODE` идет запрос к API вики, чтобы подтянуть summary статьи/термина.
- Анимация отрисовки связей (edges): линия между узлом A и узлом B постепенно окрашивается в акцентный цвет.
- UI-интерфейс презентации (кнопки Next/Prev, ползунок прогресса).

### Уровень 3: Authoring Tool (Генерация историй из Админки)
- В админ-панели (Workspace) появляется сущность "Истории" (Stories).
- Администратор может кликать на узлы графа и формировать из них линейный путь, добавляя свои комментарии к каждому шагу.
- Пользователи на фронтенде видят список доступных "туров" (например: "Введение в онтологию", "Обзор архитектуры").

## 4. Технические риски и трейдоффы
1. **Симуляция D3 vs Камера:** Во время сторителлинга физика D3 графа должна быть либо заморожена (чтобы узлы не "убегали" от камеры), либо должна быть сильно задемпфирована (alphaTarget = 0).
2. **Адаптив (Mobile):** На телефонах попапы будут перекрывать сам граф. Возможно, придется смещать фокус камеры не точно в центр, а в верхнюю/нижнюю часть экрана, чтобы оставить место для текста.
3. **Согласованность данных:** Если узел был удален из вики, история должна элегантно обрабатывать эту ошибку (пропуск шага или сообщение "Узел не найден").
