# Философия и Архитектура Gativus Wiki (Manual for LLMs)

**Внимание Исполнителю (LLM):** Перед тем как писать код, изменять архитектуру или добавлять новые системы, внимательно прочитайте этот документ. Он содержит фундаментальные принципы и технические стандарты данного проекта.

## 1. Философия проекта (Видение)

Gativus Wiki — это не просто стандартная "Википедия". Это **премиальная графовая база знаний**, построенная с вниманием к типографике, UX и эстетике.
- **Главный приоритет — Опыт Читателя (Reading Experience):** Чтение длинных текстов должно быть комфортным. Отсюда строгие сетки (golden mean margins), адаптивность, отсутствие лишнего шума и поддержка всплывающих терминов (popovers), чтобы не уводить пользователя со страницы.
- **Премиальный Минимализм:** Дизайн стремится быть похожим на дорогой продукт (SaaS документации топового уровня). Запрещены пестрые дефолтные цвета. Используется монохром (черный/белый), оттенки серого и строгий голубой/синий акцент (`sky`).
- **Связность данных (Knowledge Graph):** Данные не лежат мертвым грузом. Термины ссылаются на статьи, статьи собираются в книги, все пронизано ссылками, а для визуализации предусмотрен Граф Знаний.

## 2. Технический Стек (Узлы и Инструменты)

Мы используем современный и сверхбыстрый стек:
1. **Фреймворк:** **Nuxt 3** (Vue 3, Composition API, `<script setup lang="ts">`).
2. **Типизация:** **TypeScript**. Использование `any` допускается, но крайне нежелательно в новых компонентах.
3. **Стилизация:** **Tailwind CSS**. Вспомогательные UI-элементы реализуются через библиотеку **Nuxt UI** (компоненты `UButton`, `UInput`, `UModal` и т.д.).
4. **База данных:** Обработка идет через локальную **SQLite** БД. Движок не использует тяжелых ORM на уровне абстракций запросов. Запросы к БД осуществляются с помощью библиотеки `sqlite3` (`sqlite` / `sqlite3-promise`). Мы используем сырые SQL транзакции. Главный файл утилит работы с БД: `server/utils/db.ts` и `migrate.ts`.
5. **Глобальный стейт:** **Pinia** (стейт пользователя, токен авторизации, выбранный язык — `useLanguageStore`).

## 3. Архитектура Данных (Сущности)

Данные делятся на четкие сущности, имеющие свои таблицы в БД:
- **Articles (Статьи):** Основной лонгрид. Имеют HTML-контент, title, slug.
- **Terms (Термины/Глоссарий):** Короткие определения. Могут иметь синонимы (aliases) для авто-парсинга в тексте статей и привязку к расширенной статье (disclosure article). Поддерживают медиа: `image_url`, `video_url`, `presentation_path`.
- **Books (Книги):** Логические контейнеры/оглавления, связывающие упорядоченный список статей (через связную таблицу).
- **Categories (Категории):** Универсальные теги для классификации.

**Мультиязычность (i18n):**
В проекте реализован "плоский" подход к локализации терминов и статей. Вместо сложных EAV-таблиц атрибуты часто дублируются (`title`, `title_ru`, `title_zh` или поле `locale` в строке). Глобальный язык отслеживается в Pinia (`language` store) и пробрасывается запросами на бэкенд (`?locale=ru`).

## 4. Как реализован поиск (Search)

- Мы используем SQLite **FTS5** (Full-Text Search).
- Существует виртуальная таблица `wiki_fts`.
- Все процессы поиска автоматизированы на уровне базы данных: созданы **SQL Triggers** (на `INSERT`, `UPDATE`, `DELETE`), которые мгновенно обновляют FTS таблицу при изменении статей или терминов. Модели **запрещено** писать ручной код обновления поисковых индексов со стороны Node.js!
- Поиск доступен конечным пользователям через попап (Cmd+K).

## 5. Правила для написания кода (LLM Guidelines)

Если вам поручили задачу по этому проекту, следуйте правилам:

1. **Нет сторонних компонентов:** Если нужно сделать кнопку, вкладку, аккордеон — используйте компоненты **Nuxt UI** (пишите `UButton`, `UFormGroup` и т.д.). Не пишите свой HTML с нуля, если есть готовый компонент библиотеки. 
2. **Дизайн-консистентность:** Всегда обращайтесь к `plans/design_consistency_plan.md` перед тем, как задать цвет кнопке или закругление карточке. Округления должны быть `border-radius: 15px` для карточек и `color="black" class="rounded-xl"` для кнопок сохранения.
3. **API Endpoints (Серверная часть):** 
   - Серверные хэндлеры лежат в `server/api/...`.
   - Защищенные роуты (админка) должны проверять авторизацию. Мы используем проверки роли редактора (`userStore` + JWT токены).
4. **Хранение Медиа:**
   - Картинки из ODT, загрузки обложек книг и медиа для глоссария заливаются через эндпоинты `server/api/admin/uploads/...`.
   - Файлы складируются локально на сервере, БД сохраняет лишь путь `/uploads/cover/...`.
5. **No `v-html` danger:** В проекте используется парсинг HTML (из ODT). Будьте осторожны с XSS, но допускайте `v-html` для отрендеренных на нашем же бекенде статей.
6. **SSR & Гидратация:** Это Nuxt. Помните, что `window` и `document` существуют не сразу. Используйте `onMounted` для клиентских скриптов, взаимодействующих с DOM (например, для Lightbox в глоссарии). Не вызывайте `top-level await` в Vue компонентах в местах, которые ломают SSR.

## 6. Как анализировать баги или изменения

- **Если баг с версткой:** Сначала смотрите `pages/[page].vue`. Затем проверяйте, какие классы Tailwind применены.
- **Если баг с получением данных:** Сначала перейдите в эндпоинт `server/api/[entity]/...`, найдите `db.all` или `db.get` SQL-запрос. Зачастую проблема кроется в забытом поле в `SELECT` запросе.
- **Если нужно расширить сущность:** 
  1. Измените SQL создания таблицы в `server/utils/migrate.ts`.
  2. Обновите эндпоинты `GET` (получение) и `PUT/POST` (обновление) в `server/api/...`.
  3. Обновите интерфейс TypeScript в Vue компоненте и форму `<input>`.

**Действуй так:** "Думай медленно, пиши быстро". Проверь связи, проверь SQL запросы, используй готовый дизайн, не строй велосипедов.
